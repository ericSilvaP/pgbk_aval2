# Data Model: Complete Trip Request Creation

## Entities

### TripRequestSubmission

Client-owned input payload accepted by `POST /trip-requests`.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `requesterName` | string | Yes | Requester display name sent by the API consumer |
| `origin` | string | Yes | Departure location |
| `destination` | string | Yes | Arrival location |
| `departureAt` | string | Yes | Must be a valid date-time and is normalized to UTC before holiday validation and persistence |
| `returnAt` | string | Yes | Must be a valid date-time and is normalized to UTC before ordering validation and persistence |
| `purpose` | string | Yes | Travel purpose description |
| `passengerCount` | number | Yes | Must be greater than zero |

### Forbidden Client Fields

These fields are server-managed and are invalid in the client request.

| Field | Reason |
|-------|--------|
| `id` | Generated by the persistence layer |
| `status` | Always initialized to `pending` on creation |
| `createdAt` | Generated on the server at creation time |

### TripRequest

Persisted and returned institutional trip request.

| Field | Type | Source | Notes |
|-------|------|--------|-------|
| `id` | string | Server | Generated during persistence |
| `requesterName` | string | Client | Persisted to `requester_name` |
| `origin` | string | Client | Persisted as-is |
| `destination` | string | Client | Persisted as-is |
| `departureAt` | UTC date-time string | Client normalized by service | Stored as `DateTime`, returned as `YYYY-MM-DDTHH:mm:ss.sssZ` |
| `returnAt` | UTC date-time string | Client normalized by service | Stored as `DateTime`, returned as `YYYY-MM-DDTHH:mm:ss.sssZ` |
| `purpose` | string | Client | Persisted as-is |
| `passengerCount` | number | Client | Must be greater than zero before persistence |
| `status` | `pending` | Server | Initial lifecycle state |
| `createdAt` | UTC date-time string | Server/database | Generated during persistence, returned as `YYYY-MM-DDTHH:mm:ss.sssZ` |

### HolidayRecord

Normalized holiday data returned by `HolidaysProvider` for civil-date comparison.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `date` | string | Yes | Civil date in `YYYY-MM-DD` format |
| `name` | string | Yes | Human-readable holiday name |
| `type` | string | Optional | Provider-specific metadata preserved only if useful |

## Persistence Model

### Prisma Model: `TripRequest`

- Table: `trip_requests`
- Primary key: `id`
- Enum: `TripRequestStatus` with `pending` and `canceled`
- Column mappings:
  - `requesterName` -> `requester_name`
  - `departureAt` -> `departure_at`
  - `returnAt` -> `return_at`
  - `passengerCount` -> `passenger_count`
  - `createdAt` -> `created_at`

## State Rules

### Creation State

- New records always start with `status = pending`.
- No other state transition is part of this feature.

## Validation Order

1. Validate required request fields and reject client-managed fields.
2. Parse and normalize `departureAt` and `returnAt` to UTC.
3. Validate `returnAt >= departureAt`.
4. Validate `passengerCount > 0`.
5. Extract the normalized departure year and civil date.
6. Call `HolidaysProvider`.
7. Reject holidays or provider failure before persistence.
8. Persist only validated, non-holiday requests.

## No-Persistence Rules

The following cases MUST leave no new record in PostgreSQL:

- Request shape invalid
- `id`, `status`, or `createdAt` supplied by the client
- Invalid `departureAt` or `returnAt`
- `returnAt < departureAt`
- `passengerCount <= 0`
- Departure date is a national holiday
- `HolidaysProvider` failure
- Unexpected persistence failure before commit completion

## Relationships

- `TripRequest` has no relational dependencies in this feature.
- `HolidayRecord` is transient provider data used only for validation and is not persisted by this feature.
