# Feature Specification: Complete Trip Request Creation

**Feature Branch**: `[001-trip-request-creation]`

**Created**: 2026-06-26

**Status**: Draft

**Input**: User description: "Update the current feature to the complete, release-ready implementation of `POST /trip-requests`, including core creation, mandatory manual business validation, national holiday validation through a `HolidaysProvider` abstraction, centralized error handling, PostgreSQL persistence, standardized envelopes, and automated tests without using the real BrasilAPI."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Core trip request creation (Priority: P1)

An API consumer submits a valid institutional trip request and receives the persisted trip request with server-generated fields, normalized UTC dates, `status: "pending"`, and the standard success envelope.

**Why this priority**: This is the baseline business capability. Without successful creation, the rest of the trip request workflow does not exist.

**Independent Test**: Send a valid `POST /trip-requests` request and verify HTTP `201 Created`, the success envelope, server-generated `id` and `createdAt`, `status: "pending"`, normalized UTC dates, and a matching PostgreSQL record.

**Acceptance Scenarios**:

1. **Given** an API consumer sends `requesterName`, `origin`, `destination`, `departureAt`, `returnAt`, `purpose`, and `passengerCount` in `POST /trip-requests`, **When** the request passes all validations and holiday checks, **Then** the system persists the trip request and returns HTTP `201 Created` with `{ "success": true, "data": { ... } }`.
2. **Given** a trip request is created successfully, **When** the response body is returned, **Then** `data.id` and `data.createdAt` are generated by the server, `data.status` is `"pending"`, and `data.departureAt` and `data.returnAt` are complete ISO 8601 UTC timestamps.
3. **Given** a client includes `id`, `status`, or `createdAt` in the request body, **When** `POST /trip-requests` is submitted, **Then** the system rejects the request with HTTP `400 Bad Request`, code `VALIDATION_ERROR`, and no record is created.
4. **Given** a request reaches persistence and PostgreSQL fails unexpectedly, **When** the error is returned to the API consumer, **Then** the system responds with HTTP `500 Internal Server Error`, code `INTERNAL_SERVER_ERROR`, and no internal database details are exposed.

---

### User Story 2 - Manual business validation (Priority: P2)

An API consumer receives immediate validation feedback when local request rules fail, before the system calls the holidays provider or attempts persistence.

**Why this priority**: Mandatory local validation protects the system from invalid data and ensures that external dependencies and persistence are only used for requests that already satisfy core business rules.

**Independent Test**: Submit requests with invalid date-time values, `returnAt` earlier than `departureAt`, or `passengerCount <= 0`, and verify HTTP `400`, code `VALIDATION_ERROR`, no holidays provider call, and no persisted record.

**Acceptance Scenarios**:

1. **Given** `departureAt` or `returnAt` is not a valid date-time value, **When** `POST /trip-requests` is submitted, **Then** the system returns HTTP `400 Bad Request` with `VALIDATION_ERROR`, does not call `HolidaysProvider`, and does not persist a trip request.
2. **Given** `returnAt` is earlier than `departureAt`, **When** the request is submitted, **Then** the system returns HTTP `400 Bad Request` with `VALIDATION_ERROR`, does not call `HolidaysProvider`, and does not persist a trip request.
3. **Given** `returnAt` is equal to `departureAt`, **When** the request is submitted, **Then** the request remains eligible to continue to holiday validation and persistence.
4. **Given** `passengerCount` is zero or negative, **When** the request is submitted, **Then** the system returns HTTP `400 Bad Request` with `VALIDATION_ERROR`, does not call `HolidaysProvider`, and does not persist a trip request.

---

### User Story 3 - National holiday validation (Priority: P3)

An API consumer can only create a trip request when the normalized departure date is not a national holiday, and receives standardized error responses when the holiday provider rejects or fails.

**Why this priority**: Holiday validation is a mandatory business rule that determines whether a trip request may proceed to persistence after local validation succeeds.

**Independent Test**: Submit locally valid requests where the normalized departure date matches a national holiday or where `HolidaysProvider` fails, and verify HTTP `409` or `502`, no persisted record, and no use of the real BrasilAPI in tests.

**Acceptance Scenarios**:

1. **Given** local validation succeeds and the normalized departure civil date matches a national holiday returned by `HolidaysProvider`, **When** the request is submitted, **Then** the system returns HTTP `409 Conflict` with code `HOLIDAY_TRIP_NOT_ALLOWED` and does not persist a trip request.
2. **Given** local validation succeeds and `HolidaysProvider` fails to provide the departure year holidays, **When** the request is submitted, **Then** the system returns HTTP `502 Bad Gateway` with code `HOLIDAYS_API_UNAVAILABLE` and does not persist a trip request.
3. **Given** local validation succeeds and the normalized departure civil date is not a national holiday, **When** the request is submitted, **Then** the system continues to persistence and may return HTTP `201 Created` if no other unexpected error occurs.

### Edge Cases

- A client sends `id`, `status`, or `createdAt`; the request is rejected with `VALIDATION_ERROR` before any holidays lookup or persistence.
- `departureAt` and `returnAt` arrive with non-UTC offsets; both are normalized to complete UTC timestamps before holiday comparison and persistence.
- `returnAt` equal to `departureAt` is valid and remains eligible for holiday validation and persistence.
- `returnAt` earlier than `departureAt` is rejected locally and blocks both holidays lookup and persistence.
- `passengerCount` equal to zero is rejected locally and blocks both holidays lookup and persistence.
- `passengerCount` below zero is rejected locally and blocks both holidays lookup and persistence.
- A normalized departure date matches a national holiday; the request is rejected with `HOLIDAY_TRIP_NOT_ALLOWED` and no record is created.
- `HolidaysProvider` fails after local validation succeeds; the request is rejected with `HOLIDAYS_API_UNAVAILABLE` and no record is created.
- PostgreSQL fails after all validations pass; the request returns `INTERNAL_SERVER_ERROR` and does not expose internal details.

## Requirements *(mandatory)*

### Constitutional Requirements

- **CR-001**: This feature MUST remain backend-only and limited to the complete `POST /trip-requests` creation flow.
- **CR-002**: Success responses MUST use the standard success envelope exactly as `{ "success": true, "data": ... }`.
- **CR-003**: Error responses MUST use the centralized error envelope exactly as `{ "success": false, "error": { "code": "...", "message": "..." } }`.
- **CR-004**: `departureAt`, `returnAt`, and `createdAt` MUST be stored and returned as complete ISO 8601 UTC timestamps in `YYYY-MM-DDTHH:mm:ss.sssZ` format.
- **CR-005**: The client MUST NOT control `id`, `status`, or `createdAt`; if any of those fields are submitted, the request MUST be rejected with `VALIDATION_ERROR` before holidays lookup or persistence.
- **CR-006**: Every created trip request MUST begin with `status: "pending"`.
- **CR-007**: All mandatory creation validations MUST complete before persistence.
- **CR-008**: National holiday validation MUST occur through a `HolidaysProvider` abstraction before persistence.

### Functional Requirements

- **FR-001**: The system MUST accept trip request creation through `POST /trip-requests`.
- **FR-002**: The request body MUST require `requesterName`, `origin`, `destination`, `departureAt`, `returnAt`, `purpose`, and `passengerCount`.
- **FR-003**: The system MUST reject any request body containing `id`, `status`, or `createdAt` with HTTP `400 Bad Request` and code `VALIDATION_ERROR`.
- **FR-004**: The system MUST validate that `departureAt` and `returnAt` are valid date-time values before calling `HolidaysProvider` or the repository.
- **FR-005**: The system MUST normalize `departureAt` and `returnAt` to complete ISO 8601 UTC format before holiday validation, persistence, and response mapping.
- **FR-006**: The system MUST validate that `returnAt` is equal to or later than `departureAt` after date normalization and MUST reject invalid ordering with HTTP `400 Bad Request` and code `VALIDATION_ERROR`.
- **FR-007**: The system MUST validate that `passengerCount` is greater than zero and MUST reject zero or negative values with HTTP `400 Bad Request` and code `VALIDATION_ERROR`.
- **FR-008**: When any local validation fails, the system MUST NOT call `HolidaysProvider`, MUST NOT call the repository, and MUST NOT persist a trip request.
- **FR-009**: After local validation succeeds, the system MUST extract the normalized departure year and civil date in `YYYY-MM-DD` format and MUST call the `HolidaysProvider` abstraction before persistence.
- **FR-010**: If the normalized departure civil date matches a national holiday returned by `HolidaysProvider`, the system MUST reject creation with HTTP `409 Conflict` and code `HOLIDAY_TRIP_NOT_ALLOWED`.
- **FR-011**: If `HolidaysProvider` fails after local validation succeeds, the system MUST reject creation with HTTP `502 Bad Gateway` and code `HOLIDAYS_API_UNAVAILABLE`.
- **FR-012**: When holiday validation fails or the provider fails, the system MUST NOT persist a trip request.
- **FR-013**: Only when the request passes local validation and holiday validation MAY the system persist the trip request in PostgreSQL.
- **FR-014**: On successful persistence, the system MUST generate a unique `id` and server-generated `createdAt` value, set `status` to `"pending"`, and return HTTP `201 Created`.
- **FR-015**: A successful response MUST return `data` containing `id`, `requesterName`, `origin`, `destination`, `departureAt`, `returnAt`, `purpose`, `passengerCount`, `status`, and `createdAt`.
- **FR-016**: If PostgreSQL persistence fails unexpectedly after all validations pass, the system MUST return HTTP `500 Internal Server Error` with code `INTERNAL_SERVER_ERROR` and MUST NOT expose internal database messages, SQL statements, stack traces, connection strings, or file paths.
- **FR-017**: The system MUST preserve the intended application flow in this order: receive the HTTP request; validate request fields; reject client-managed fields; validate and normalize dates; validate `returnAt >= departureAt`; validate `passengerCount > 0`; call `HolidaysProvider`; reject national holiday departures; persist with `status: "pending"`; return HTTP `201 Created`.

### Key Entities *(include if feature involves data)*

- **TripRequestSubmission**: The client-provided payload containing `requesterName`, `origin`, `destination`, `departureAt`, `returnAt`, `purpose`, and `passengerCount`.
- **TripRequest**: The persisted institutional trip request containing `id`, `requesterName`, `origin`, `destination`, `departureAt`, `returnAt`, `purpose`, `passengerCount`, `status`, and `createdAt`.
- **HolidayRecord**: A normalized national holiday entry exposed by `HolidaysProvider`, containing at least the holiday civil date and descriptive metadata needed for comparison.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of valid `POST /trip-requests` submissions return HTTP `201 Created` with the standard success envelope and all required response fields.
- **SC-002**: 100% of successful creations return `departureAt`, `returnAt`, and `createdAt` as complete ISO 8601 UTC timestamps and persist matching values in PostgreSQL.
- **SC-003**: 100% of requests rejected by local validation, holiday validation, or holidays provider failure result in no persisted trip request record.
- **SC-004**: 100% of national holiday departures return HTTP `409 Conflict` with code `HOLIDAY_TRIP_NOT_ALLOWED`, and 100% of holidays provider failures return HTTP `502 Bad Gateway` with code `HOLIDAYS_API_UNAVAILABLE`.
- **SC-005**: 100% of unexpected persistence failures during creation return the centralized `INTERNAL_SERVER_ERROR` response without exposing internal implementation details.

## Assumptions

- The API consumer sends JSON requests to `POST /trip-requests`.
- `HolidaysProvider` is the application abstraction responsible for obtaining national holidays from the configured external source.
- Automated tests use a test double for `HolidaysProvider` and MUST NOT access the real BrasilAPI.
- Authentication, authorization, frontend, vehicle allocation, driver allocation, approvals, and other non-creation workflows remain out of scope for this feature.
